# General Working Instructions for Claude Code (use with every project)

These are standing rules that apply on top of whatever project-specific spec document is provided alongside this file. Follow both together.

## 1. Code & Commit Style
- Write code and commit messages as a normal developer would — no comments, docstrings, commit messages, or file headers that mention Claude, AI, or any AI tool.
- Do not add phrases like "Generated by AI," "AI-assisted," "Co-authored-by: Claude," or similar anywhere in code, commit messages, README, or docs.
- Commit messages should be clear, conventional, and describe *what changed and why* — written the way a developer would normally write them (e.g. `feat: add xG model training pipeline`, `fix: correct shot angle calculation`).
- Keep commit history clean and logical — one meaningful change per commit, not giant dumps.

## 2. Repository Setup
- Work inside the local project repository/folder that will be connected.
- Follow standard repo structure conventions for the language/framework being used (e.g. `src/`, `tests/`, `docs/`, `README.md`, `.gitignore`).
- Include a proper `.gitignore` for the stack in use (don't commit virtual envs, model checkpoints, node_modules, `.env` files, etc.).

## 3. Documentation
- Maintain a `README.md` that explains: what the project does, how to set it up, how to run it, and current status/progress.
- As features are built, keep documentation updated in the same commit/PR as the code change — not as an afterthought.
- Any non-obvious design decision should get a short explanation (in code comments or a `docs/` note) — written plainly, as a developer's own reasoning, not as an AI disclaimer.
- Explain technical concepts and choices in plain language as you go, so I can follow and learn from the process — but keep this explanation in chat/conversation, not embedded as AI-attribution in the codebase itself.

## 4. Testing
- Write tests alongside features, not only at the end.
- Cover: core logic (e.g. model input/output shapes, key functions), edge cases, and at least one basic integration/end-to-end check per major feature.
- Tell me clearly what is tested and what isn't yet, so nothing is assumed done when it's untested.

## 5. Workflow
- Build incrementally, in the phases defined by the project-specific spec document.
- After each phase/milestone, stop and summarize what was built, what was tested, and what's left — don't silently continue to the next phase.
- If something in the spec is ambiguous, ask before assuming.

## 6. Ownership
- This is my own independent project. All code, documentation, and commits should read as my own work product.
- Do not include any external tool attribution anywhere in the deliverable (code, comments, commits, README, docs).
